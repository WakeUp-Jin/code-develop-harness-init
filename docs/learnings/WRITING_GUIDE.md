# 学习文档编写指南

触发学习记录后，读这份文档来决定格式和写法。

## 格式选择

### HTML 可视化教程（主力格式）

适合有深度、需要图示或代码对比的内容。要求：

- **单文件**，浏览器直接打开，不依赖外部资源。
- 内嵌 CSS + 少量 JS（代码高亮、Tab 切换等），确保离线可用。
- 建议包含以下区块（不强制，自行判断取舍）：
  - **是什么**：一段话讲清概念。
  - **为什么需要**：问题场景 + 没用这个方案时的痛点。
  - **怎么用**：代码示例，Before/After 对比。
  - **核心要点**：3–5 条关键 takeaway。
  - **常见陷阱**：容易踩坑的地方。
  - **自检问题**：2–3 个检验理解的问题。

### Markdown 速记（轻量补充）

适合一页纸能讲清的内容：某个 API 的关键注意事项、一个小技巧、对外部文章的摘要。

## 写作原则

- 读者是**开发者本人**，不是 Agent。语言要教学化，不是记录化。
- 每篇聚焦**一个核心知识点**，宁可拆成两篇也不要混在一起。
- 重点是**为什么**和**怎么想**，不要复述代码文档。
- 可以关联对应的 history 文件，说明"这个知识点是从哪次变更中提炼的"。

## 典型场景

1. **使用了设计模式** — 为什么这里适合策略模式而不是 if-else？
2. **非显而易见的 bug 修复** — 闭包陷阱、竞态条件、内存泄漏背后的原理。
3. **架构决策与取舍** — 为什么选 SSR 不选 CSR？为什么用事件驱动而不是轮询？
4. **新技术/库的引入** — 第一次用某个框架或工具链，核心概念值得整理。
5. **重构揭示的本质** — 重构暴露出来的耦合问题、职责混乱，是理解好设计的最佳素材。
6. **性能优化的原理** — 不只是"加了缓存变快了"，而是缓存策略、失效机制、时空权衡。

## 不值得单独记的

以下内容在 history 里顺带提一句即可：

- 纯 CRUD、简单配置修改。
- 重命名、格式调整、依赖升级（除非涉及 breaking change 的应对策略）。
- 项目特有的业务逻辑（除非背后有通用的建模思路）。
