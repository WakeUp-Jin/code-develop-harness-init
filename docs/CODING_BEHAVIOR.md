# 编码行为纪律

这份文档约束编码过程中的具体行为模式，对人和 Agent 都适用。
灵感来源：Karpathy 对 LLM 编码常见失误的观察。

## 先想清楚再动手

- 动手之前，显式列出你的假设。不确定的就问，不要默默选一个。
- 如果存在多种理解，把选项列出来让人选，不要私自决定。
- 如果有更简单的方案，主动说出来。该推回去就推回去。
- 遇到不清楚的地方，停下来，说出哪里不清楚，然后问。

## 简洁优先

- 不做没被要求的功能。不做单次使用代码的抽象。不加投机性的"灵活配置"。
- 不为不可能发生的场景写错误处理。
- 如果 200 行能缩到 50 行，重写。
- 判断标准：一个资深工程师会不会说"这太复杂了"？会的话，简化。

## 外科手术式修改

- 只改需要改的。不顺手"改进"相邻代码、注释或格式。
- 不重构没出问题的东西。
- 匹配已有代码风格，即使你会用不同方式写。
- 你的改动产生的废弃 import/变量/函数，必须清理；原来就存在的死代码，提一嘴但不要动。
- 检验方式：每一行变更都应能直接追溯到用户的需求。

## 目标驱动执行

- 把任务转化为可验证的目标，而不是模糊的描述。
  - "加验证" → "写无效输入的测试，然后让它通过"
  - "修 bug" → "写复现测试，然后让它通过"
  - "重构 X" → "确保重构前后测试都过"
- 多步任务，用 "step → verify" 格式落计划，每步附带检查项。
- 强验收标准让你能独立循环推进；弱标准（"让它能跑"）会导致反复确认。
